Ecommerce - Guia de despliegue del Frontend en Ubuntu sin Docker (Contabo)

Fecha: 2026-07-13
Alcance: Solo Frontend Next.js en VPS Ubuntu + Nginx + HTTPS (Let's Encrypt) + API en VPS Windows separado.

Arquitectura objetivo
- VPS Windows: backend API ASP.NET Core (ya operativo).
- VPS Ubuntu (Contabo): frontend Next.js 15 sin Docker.
- Dominio frontend ejemplo: tienda.midominio.com.
- Dominio API ejemplo: api.midominio.com.

1) Requisitos previos
- Ubuntu 22.04/24.04 con IP publica.
- Dominio/subdominio apuntando al VPS Ubuntu.
- API del backend accesible por HTTPS desde internet.
- Puertos 22, 80 y 443 abiertos en firewall/security group.

2) Preparar servidor Ubuntu
Ejecutar:
  sudo apt update && sudo apt upgrade -y
  sudo apt install -y ca-certificates curl gnupg unzip git nginx

Configurar firewall:
  sudo ufw allow OpenSSH
  sudo ufw allow 80
  sudo ufw allow 443
  sudo ufw enable

3) Instalar Node.js LTS (sin Docker)
Recomendado Node 20 LTS para Next.js 15.

  curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
  sudo apt install -y nodejs
  node -v
  npm -v

4) DNS del frontend
En tu proveedor DNS crea registros A:
- tienda.midominio.com -> IP_PUBLICA_VPS_UBUNTU

Si usaras raiz y www:
- midominio.com -> IP_PUBLICA_VPS_UBUNTU
- www.midominio.com -> IP_PUBLICA_VPS_UBUNTU

Validar:
  nslookup tienda.midominio.com

5) Publicar codigo en el VPS Ubuntu
Ejemplo en /var/www:
  sudo mkdir -p /var/www
  cd /var/www
  sudo git clone https://github.com/jfernandez96/ecommerce.git ecommerce
  sudo chown -R $USER:$USER /var/www/ecommerce
  cd /var/www/ecommerce/src/frontend

6) Variables de entorno de produccion del frontend
Crear archivo .env.production en src/frontend:

  NEXT_PUBLIC_API_URL=https://api.midominio.com/api/v1
  NEXT_PUBLIC_SITE_URL=https://tienda.midominio.com
  NODE_ENV=production

Notas importantes:
- NEXT_PUBLIC_API_URL debe apuntar al backend Windows real (no localhost).
- Si tu API usa cookies/sesion, valida CORS y cookies secure/samesite en backend.

7) Build y arranque local del frontend (puerto interno 3000)
Desde /var/www/ecommerce/src/frontend:
cd /var/www/ecommerce/src/frontend

  npm ci
  npm run build
  npm run start -- -p 3000

Probar desde el servidor:
  curl http://127.0.0.1:3000

8) Ejecutar como servicio con systemd (recomendado)
Crear servicio:
  sudo nano /etc/systemd/system/ecommerce-frontend.service

Contenido:

[Unit]
Description=Ecommerce Frontend Next.js
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/var/www/ecommerce/src/frontend
Environment=NODE_ENV=production
EnvironmentFile=/var/www/ecommerce/src/frontend/.env.production
ExecStart=/usr/bin/npm run start -- -p 3000
Restart=always
RestartSec=5
KillSignal=SIGINT
SyslogIdentifier=ecommerce-frontend

[Install]
WantedBy=multi-user.target

Ajustar permisos para que www-data pueda leer el proyecto:
  sudo chown -R www-data:www-data /var/www/ecommerce

Habilitar e iniciar:
  sudo systemctl daemon-reload
  sudo systemctl enable ecommerce-frontend
  sudo systemctl start ecommerce-frontend
  sudo systemctl status ecommerce-frontend

Logs del servicio:
  sudo journalctl -u ecommerce-frontend -f
9) Configurar Nginx reverse proxy
Crear archivo:
  sudo nano /etc/nginx/sites-available/ecommerce-frontend

Contenido HTTP inicial:

server {
    listen 80;
    server_name tienda.midominio.com;
  client_max_body_size 25M;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}

Nota importante para uploads de banners/promociones/productos:
- Si Nginx no tiene client_max_body_size, devolvera 413 Request Entity Too Large (default ~1MB).
- Mantener el mismo valor en el server HTTPS generado por Certbot (archivo en /etc/nginx/sites-available/ecommerce-frontend).

Validar y recargar:
sudo nginx -t
sudo systemctl reload nginx
sudo systemctl restart ecommerce-frontend

Activar sitio:
  sudo ln -s /etc/nginx/sites-available/ecommerce-frontend /etc/nginx/sites-enabled/ecommerce-frontend
  sudo nginx -t
  sudo systemctl reload nginx

10) SSL con Let's Encrypt (Certbot)
Instalar Certbot:
  sudo apt install -y certbot python3-certbot-nginx

Emitir certificado y forzar redireccion HTTPS:
  sudo certbot --nginx -d tiendaonline.softcoretechnology.com.pe --redirect -m jfernandez@softcoretechnology.com.pe --agree-tos --no-eff-email

Probar renovacion:
  sudo certbot renew --dry-run

Ver timer:
  systemctl status certbot.timer

11) Validaciones post-deploy
- https://tienda.midominio.com responde.
- https://tienda.midominio.com/admin/login responde.
- El frontend consume datos desde https://api.midominio.com/api/v1.
- No hay errores CORS en navegador.
- Revisar logs:
  sudo journalctl -u ecommerce-frontend -f
  sudo journalctl -u nginx -f

12) Flujo de actualizacion (sin Docker)
Desde /var/www/ecommerce:
  cd /var/www/ecommerce
  git pull
  cd src/frontend
  npm ci
  npm run build
  sudo systemctl restart ecommerce-frontend
  sudo systemctl status ecommerce-frontend

13) Troubleshooting rapido
- Error 502 Bad Gateway:
  Verifica que el servicio ecommerce-frontend este arriba y escuchando en 3000.
  Comandos:
    sudo systemctl status ecommerce-frontend
    sudo journalctl -u ecommerce-frontend -n 100

- Error de CORS en frontend:
  Ajusta CORS en backend Windows para permitir https://tienda.midominio.com.

- Variables NEXT_PUBLIC no reflejan cambios:
  Recuerda que NEXT_PUBLIC se inyecta en build. Ejecuta npm run build de nuevo y reinicia servicio.

- SSL no emite:
  Verifica DNS apuntando al VPS Ubuntu y puerto 80 abierto.

- Error 405 (HTTP verb not allowed) en produccion al editar/eliminar:
  Suele ocurrir en IIS del backend Windows por WebDAV bloqueando PUT/PATCH/DELETE.
  Acciones:
    1. Deshabilitar WebDAV en el sitio de la API.
    2. Publicar incluyendo web.config con remove de WebDAV y handler aspNetCore verb="*".
    3. Reiniciar sitio o App Pool en IIS.

14) Checklist minimo de seguridad
- Backend con Jwt__Secret robusto y rotacion controlada.
- Swagger restringido o deshabilitado en produccion.
- Password admin seed cambiado.
- Backups de SQL Server validados.
- Monitoreo de errores 5xx en frontend y backend.
